# Migration: v0.2 → v0.3

v0.3 introduces the "infinite grouping" feature surface: multi-level
`groupBy`, multi-field `sort`, `showAllGroups`, and cross-group drag-drop
via `onEventGroupChange`.  It is **fully backwards-compatible** — v0.2
consumers upgrade with no code changes.

---

## New props on `<WorksCalendar>`

| Prop                 | What it does                                                                     |
| -------------------- | -------------------------------------------------------------------------------- |
| `groupBy`            | 1-, 2-, or 3-level grouping. Accepts string, string[], or `GroupConfig[]`.       |
| `sort`               | `SortConfig \| SortConfig[]` — ordered tiebreakers within each group.            |
| `showAllGroups`      | Surface empty groups + cross-group copies.                                       |
| `onEventGroupChange` | `(event, patch) => void` — fires on cross-group / cross-row DnD.                 |

Full reference: [`GROUPING_API.md`](./GROUPING_API.md).

---

## Saved-view schema: v2 → v3

Saved views now persist grouping state alongside filters.

**v2 shape** (still readable by v0.3):

```json
{
  "version": 2,
  "name":    "Night shift",
  "filters": { /* ... */ }
}
```

**v3 shape** (emitted by v0.3):

```json
{
  "version": 3,
  "name":    "Night shift",
  "filters": { /* ... */ },
  "groupBy":        ["role", "shift"],
  "sort":           [{ "field": "start", "direction": "asc" }],
  "showAllGroups":  false
}
```

### Behaviour

- v0.3 **reads** both v2 and v3. v2 views load with `groupBy`, `sort`,
  `showAllGroups` set to `null` / `false` (i.e. no grouping).
- v0.3 **writes** v3 for all newly-saved views.
- Downgrading to v0.2 silently ignores the new fields (they drop through the
  v2 whitelist); filter state is preserved.
- The whitelist validates `groupBy` shape, `sort` field types, and booleans.
  Unrecognised keys are stripped — no arbitrary data leaks into localStorage.

No migration script is required.  Existing persisted v2 entries are still
valid; next save promotes them to v3.

## New engine op: `group-change`

The `CalendarEngine` gains a new operation type for field-only mutations
(i.e. reassigning an event between resources, categories, roles):

```ts
{
  type:   'group-change',
  id:     string,
  patch:  Partial<Omit<EngineEvent, 'id' | 'start' | 'end'>>,
  source?: OperationSource,
}
```

- Time fields (`start`, `end`) are explicitly **not** mutable through this
  op — use `move` / `resize` for that.
- Validators run through a dedicated `groupChangeValidators` chain on
  `OperationContext`, so domain rules (role-match, shift-count limits,
  on-call ownership) can accept / soft-warn / hard-reject reassignments
  using the same protocol as existing ops.
- Undo/redo works identically to other ops.

### Validator signature

```ts
type GroupChangeRule = (
  change: { event: EngineEvent; patch: Readonly<Record<string, unknown>> },
  ctx:    OperationContext,
) => Violation | null;
```

Register by passing `{ groupChangeValidators: [...] }` when constructing the
engine context.  No validator = accept-all (backwards-compatible default).

## DnD callback contract

When `onEventGroupChange` is supplied:

- Events become draggable across groups (AgendaView leaves) and across rows
  (TimelineView resource rows).
- Dropping an event produces a patch shaped by the drop target:
  - **Timeline row**: `{ resource: <employeeId or null> }`.
  - **Agenda leaf group**: one field per `groupBy` dimension the drop crosses.
- Same-group / same-row drops are short-circuited — the callback does not
  fire, and no engine op is dispatched.

When `onEventGroupChange` is absent, DnD is disabled (events are not
`draggable`) — maintaining the v0.2 behaviour for consumers that don't opt in.

## Upgrade checklist

- [ ] No code changes needed if you don't want grouping.
- [ ] To enable grouping: pass `groupBy` as a string (single-level) or
      string[]/`GroupConfig[]` (nested).
- [ ] To enable cross-group DnD: provide `onEventGroupChange`.  Persist the
      patch with your own save logic, or fall back to the engine's default
      which emits `save(event)` on the updated event.
- [ ] To persist per-user grouping: no changes — saved views already round-trip
      the new fields in schema v3.
- [ ] To enforce domain rules on reassignment: register a
      `GroupChangeRule` via `OperationContext.groupChangeValidators`.
- [ ] If your tests stub `EngineOperation`, add the `'group-change'` variant to
      your union or exhaustiveness checks.

---

## Breaking changes

None.  v0.3 is additive.
